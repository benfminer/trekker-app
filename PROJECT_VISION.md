## 1. Project Overview

TRACE Trekkers is a year-long collective fitness challenge for the entire TRACE school community — students, teachers, and staff — pooling their miles and steps to walk around the world, starting and ending in San Diego, CA (~24,900 miles total). The app replaces a Google Form + spreadsheet system with a purpose-built experience: a clean, frictionless log entry page and a bold, adventurous live map that makes the group's collective progress feel real, exciting, and worth coming back to.

## 2. Goals & Success Metrics

- **Primary goal:** Keep the whole school community engaged in the initiative consistently throughout the year — not just at the start
- **Secondary goals:**
  - Make collective progress visible and exciting in real time
  - Reduce friction for submitting miles/steps
  - Celebrate milestones (countries, continents, oceans, capitals) as they're reached
- **How success will be measured:** Sustained submission activity throughout the school year; community engagement with the map; completion of the circumnavigation route

## 3. Target Users

The entire TRACE school community — adult students, teachers, and staff at TRACE in San Diego, CA. Users are tech and phone-savvy. They submit individually or as a group/class. No accounts or login required — anyone with the link can participate. The experience should feel empowering and celebratory.

## 4. Core Features — v1 Scope

### Must Have
- Open-access submission form (no login) — log miles or steps, individually or as a group/class
- Automatic step-to-mile conversion at 2,500 steps per mile
- Live map showing the great circle route from San Diego around the world, with a marker at the current collective position
- Milestone system — triggered when the group enters a new country, crosses a sea or ocean, enters a new continent, or reaches a major city or capital
- Milestone celebration UI — something visually rewarding when a milestone is hit
- Migration of all historical data (~7,135 miles) from previous Google Form/CSV exports, with support for additional data imports later
- Admin interface to review, edit, and delete submissions (manage bad or duplicate entries)

### Out of Scope (v1)
- User accounts or login
- Personal activity logs or individual progress tracking
- Leaderboards or competitive elements
- Manually curated landmark milestones beyond the automatic system (TBD for later)

## 5. User Journey

1. User opens the app on their phone or computer — no login required
2. They see the live map with the current position of the group on the route
3. They tap/click "Log Activity" to go to the submission page
4. They enter their name or class name, the date, and choose miles or steps — app converts automatically
5. They submit — confirmation is shown
6. The map updates to reflect the new total
7. If a milestone is crossed, a celebration moment triggers on the map

**Admin flow:**
1. Admin navigates to a protected admin route (simple password or basic auth)
2. Views all submissions in a table
3. Can edit or delete bad/duplicate entries

## 6. Design & UX Direction

### Visual Personality
Two distinct experiences within one app:

- **Log entry page:** Clean, minimal, calm. Easy to use on a phone. No distractions. Fast.
- **Live map page:** Bold, adventurous, expedition-like. The map is the hero. Orange and black. Feels alive.

The overall product should feel empowering and celebratory — not clinical, not childish. It should feel like something worth showing off.

### Key UI Moments
- The map view with the route and current position marker — this is the heart of the app
- Milestone celebration — a satisfying moment when the group crosses into a new country, continent, or body of water
- The submission confirmation — should feel like a contribution, not just a form receipt

### Design References
- Bold adventure/expedition map aesthetics
- School colors: Orange (#FF6B00 or similar) and Black — used confidently, not sparingly
- Clean form design: single-column, large tap targets, minimal labels

### Inspiration URLs
- To be added to `/inspiration/` folder

### Mobile vs Desktop
Mobile-first. The log entry page should be thumb-friendly. The map should scale gracefully to desktop where it can be more immersive.

### Accessibility
Standard accessibility practices: sufficient color contrast, large tap targets, readable font sizes. No specialized accommodations beyond standard best practices.

## 7. Brand Voice & Tone

### Personality
Energetic, collective, celebratory, clear, encouraging

## 9. Technical Specification

### Stack
- **Frontend:** React + TypeScript + Tailwind CSS
- **Backend:** Ruby on Rails (API mode)
- **Database:** PostgreSQL
- **Auth:** Simple admin password protection (basic auth or a single hardcoded admin route — no full auth system needed)
- **Map library:** Mapbox GL JS (recommended for quality and route control) or Leaflet as fallback
- **Hosting:** AWS (Elastic Beanstalk or EC2 for Rails; S3/CloudFront for frontend assets)
- **Third-party integrations:** Mapbox API (map tiles and route rendering)

### Constraints
- Solo build with AI agents
- 1–2 week launch target
- Must support CSV data import for historical data and future bulk imports

### Performance & Security Requirements
- Map should load fast on mobile
- Admin route should be protected — not publicly discoverable
- No PII collected beyond names (which may be class names, not individuals)

## 10. Content & Data Model

**Core data:**
- Submissions: name/class name, date, input type (miles or steps), input value, converted miles, timestamp
- Milestones: type (country, continent, ocean, city/capital), name, mile marker, triggered (boolean), triggered date
- Route: pre-computed great circle path from San Diego around the world (~24,900 miles), stored as a series of coordinates

**Data flow:**
- Users submit via open form → stored in PostgreSQL
- Total miles calculated from all submissions → used to compute current position on route
- Position checked against milestone markers → milestone triggered if threshold crossed
- Admin can view, edit, delete any submission

**Historical data:**
- ~7,135 miles from previous Google Form CSVs to be migrated at launch
- Support for additional CSV imports in the future
